## 🧭 الإجراء لكل محادثة (اقرأه قبل ما تبدأ)

**استراتيجية الدمج: تدريجي** — كل دفعة = فرع خاص + PR صغير يُدمج فور التحقق. لا تنتظر باقي الدفعات.

كل محادثة تأخذ **دفعة واحدة فقط** من الجدول تحت (التي حالتها `⬜ لم تبدأ`)، وتنفّذ:

0. **افتح فرع خاص بالدفعة** من `main` المحدّث، مثلاً: `git checkout main && git pull && git checkout -b claude/image-audit-batch-<رقم>`.
1. **حدّث حالة الدفعة** إلى `🚧 جارية` في جدول التقدّم + اكتب رقم المحادثة/التاريخ.
2. لكل منتج في الدفعة:
   - دوّر على الموديل على الموقع الرسمي: **https://en.tiandy.com** (ابحث بالـ SKU أو وصفه). إن لم تجده مباشرة، استعمل بحث ويب: `Tiandy <SKU>` لإيجاد صفحة المنتج الرسمية أو موزّع معتمد (tiandy.systems / forbatt / a2zsecuritycameras).
   - **حمّل الصورة الحالية والصورة الرسمية وقارنهما بصرياً** (`curl` ثم أداة Read على ملف الصورة) — تأكد أن **الشكل (bullet/turret/dome/PTZ/box) ونوع المنتج يطابقان** الموديل. لا تكتفِ بالنص.
   - إذا الصورة الحالية **صحيحة** → أشّر ✅ بدون تغيير.
   - إذا **خاطئة** → خذ رابط الصورة الرسمية من CDN تياندي (`https://bsg-i.nbxc.com/product/...`)، وحدّث:
     - **قاعدة البيانات الحيّة:** `update store.products set image_url='...', updated_at=now() where sku='...';`
     - **ملف الـ seed:** `supabase/seed/0002_store_data.sql` (نفس الرابط).
   - عبّي عمود «الصورة الصحيحة» + أشّر الحالة (✅ صحيحة / 🔁 صُحّحت).
3. بعد إنهاء الدفعة: `commit` بكل التغييرات (هذا الملف + seed) + حدّث جدول التقدّم إلى `✅ مكتملة`، ثم `push` الفرع، **افتح PR صغير، وادمجه فوراً** بعد التحقق (الدفعة مستقلة وقصيرة). الكتالوج الحيّ محدّث أصلاً من قاعدة البيانات.
4. **لا تأخذ أكثر من دفعة بالمحادثة** — هيك الفحص يضل دقيق ومركّز، وكل PR صغير وسهل المراجعة.
